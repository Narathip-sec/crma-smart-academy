2. Product Purpose

CRMA Smart Academy should become a one-stop digital service for cadets, distributed through the CRMA LINE Official Account as a LIFF app. It should replace scattered LINE groups, whiteboards, spreadsheets, and manual announcements with a controlled, auditable, role-based platform.

Cadets

Consume schedules, grades, daily tasks, meals, activity rooms, notifications, reports, and personal development guidance.

Instructors

Publish class data, record or verify grades, manage assignments and training-related events.

Command / Admin

Manage cadet data, official announcements, calendars, reports, audit logs, and operational dashboards.

5. Backend Systems Needed

Identity & Access

LIFF profile, LINE user ID binding, @crma.ac.th email verification, role-based access control, user provisioning, account deactivation.

  • Tables: User, CadetProfile, RoleAssignment, UserLineIdentity.
  • Services: LIFF login callback, email-domain allowlist, session/JWT, admin invite/import.

Academic Core

Schedules, courses, enrollments, grade records, semester lock rules, GPA/GPAX calculation, current-term visibility policy.

  • Tables: Term, Course, Enrollment, ScheduleEntry, GradeRecord.
  • APIs: getMySchedule, getMyGrades, getTermAvailability.

Announcements & Content

News, events, hero carousel items, world news links, service pages, rich media, audience targeting.

  • Tables: Announcement, ContentAsset, AudienceRule, ReadReceipt.
  • Admin: create/publish/schedule/archive, attach image, target battalion/year/company.

Calendar & Tasks

Academic/military/personal events, recurrence, task assignment, daily orders, completion tracking, reminders.

  • Tables: CalendarEvent, EventAudience, Task, TaskAssignment, TaskCompletion.
  • Rules: official vs personal events, who can create, who can mark done.

Meal & Health

Mess hall menu, nutrition, canteen additions, AI recommendations, PFT data, wearable sync connections, weather data.

  • Tables: MealMenu, FoodItem, FoodLog, FitnessMetric, PftResult, HealthConnection, ConsentRecord.
  • Risk: health and nutrition data require explicit consent and minimization.

Report/Fix Workflow

Maintenance reports, emergency channel distinction, map coordinates, photo uploads, status workflow, technician assignment.

  • Tables: ReportTicket, TicketPhoto, TicketStatusEvent, MaintenanceTeam.
  • Storage: signed upload URLs, virus scanning, retention policy.

LINE Messaging

Push, narrowcast, multicast, Flex messages, RSVP links, service reminders, repair status notifications.

  • Services: Messaging API worker, webhook receiver, idempotency, retry queue.
  • Tables: Notification, NotificationDelivery, LineWebhookEvent.

Audit, Compliance, Operations

PDPA controls, audit logs, consent, data-subject requests, admin actions, backups, monitoring, incident response.

  • Tables: AuditLog, ConsentRecord, DataSubjectRequest, AdminAction.
  • Ops: Vercel logs, database backups, alerting, runbooks, DPO workflow.

6. Recommended Architecture

LINE OA / Rich Menu / Flex Message │ ▼ LINE LIFF app on Vercel Next.js App Router · React · TypeScript · Tailwind │ ▼ Next.js API layer / server actions Auth · RBAC · validation · audit middleware │ ├── PostgreSQL via Prisma ├── Object storage for report/news images ├── Queue/worker for LINE push delivery ├── Optional AI service for recommendations/Q&A └── Future NetworkX service for activity/social analysis

Recommendation

Keep the first production version monolithic inside Next.js where practical: app, API routes, Prisma, auth, and audit middleware in one deployable system. Add separate workers only for push notification delivery, image processing, and future analytics once traffic and ownership justify it.

9. Detailed Delivery Plan

Confirm Product Boundaries

Freeze Phase 1 scope: Home, Profile/Grades, Class Schedule, Calendar, LINE login, admin seed imports, and basic push notifications.

  • Approve screen list and labels: especially To-do-list vs To-do.
  • Confirm POC audience: one pilot company, approximate cadet count, admin roles.
  • Approve production language behavior: English default, TH/EN preference stored per user.

Stabilize Design System

Move prototype tokens into production components, remove prototype-only artifacts, and make light/dark mode accessible.

  • Use CSS variables from the prototype palette.
  • Replace emoji icons with a consistent icon system where possible.
  • Move DEV language/theme toggle into Settings/Profile before production.
  • Check Thai text wrapping on mobile widths.

Build Auth + User Foundation

Initialize LIFF, bind LINE user ID to a CRMA email/cadet record, and enforce role-based access.

  • Implement LIFF init and login redirect handling.
  • Add email verification or admin-imported invitation flow.
  • Reject non-CRMA domains and log rejected attempts.
  • Add audit middleware for sensitive reads and writes.

Expand Database Schema

Extend the existing Prisma Phase 1 schema into a system that supports content, calendar, notifications, reports, and preferences.

  • Add User, RoleAssignment, Term, Announcement, CalendarEvent, UserPreference.
  • Add Notification and NotificationDelivery before LINE push work.
  • Add ReportTicket and TicketStatusEvent if Report/Fix enters Phase 1.
  • Write seed data matching the prototype so UI can be tested without academy integrations.

Port Remaining Prototype Screens

Replace placeholder pages with production React components, starting with the highest-value Phase 1 pages.

  • Calendar: month grid, selected-day agenda, event dots with overflow cap.
  • To-do-list: categories, filters, completion, urgent marker.
  • Service: search, frequent services, tile grid, routing to completed screens.
  • Later: Meal, Health AI, Report/Fix, Activity with real workflow decisions.

Build Admin Data Inputs

Production cannot rely on mock data. Add controlled import and publishing flows before pilot usage.

  • CSV import for cadets, courses, enrollments, and schedules.
  • Admin content editor for news/events and hero carousel.
  • Grade import with completed-semester lock validation.
  • Preview mode for announcements and LINE messages.

Integrate LINE Messaging

Use push/narrowcast carefully; every notification should be targeted, logged, and user-relevant.

  • Create rich menu links to Home, Calendar, To-do-list, Activity, Service.
  • Add webhook receiver for postback events if Flex messages are used.
  • Add queue/retry for delivery and idempotency keys.
  • Expose admin delivery reports.

Compliance Review Before Pilot

PDPA and military context must be handled before real cadet data enters the system.

  • Publish privacy notice and consent text in Thai and English.
  • Confirm Thailand data residency.
  • Confirm no classified military data in the POC.
  • Verify audit logs for grade access, profile access, reports, admin actions, and notifications.

Pilot, Measure, Iterate

Run with the pilot company, measure usage, gather cadet/admin feedback, and decide Phase 2 priorities.

  • Track daily active users, screen completion, notification open rates, report SLA, and schedule/grade lookup frequency.
  • Collect structured feedback inside the LIFF app.
  • Hold admin review after two weeks and cadet review after one month.
  • Promote Phase 2 modules based on usage, not just prototype appeal.

10. Is the Prototype Sufficient?

Sufficient For

  • Visual direction and mobile density.
  • Main navigation and screen inventory.
  • Bilingual and light/dark behavior.
  • POC demo conversations with stakeholders.
  • Seed data shape for first UI port.

Not Sufficient For

  • Production data model completeness.
  • RBAC, PDPA, audit logging, retention, and security controls.
  • Admin workflows and source-of-truth data ingestion.
  • LINE webhook, Messaging API delivery, and retry behavior.
  • Health/AI consent, wearable provider contracts, and medical boundary rules.
