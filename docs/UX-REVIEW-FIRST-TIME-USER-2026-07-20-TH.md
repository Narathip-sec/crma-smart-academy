# รีวิว UX/UI มุมมองผู้ใช้ครั้งแรก — 2026-07-20

**วิธีทดสอบ:** จำลองเป็น นนร. เปิดแอปครั้งแรก ไล่ครบทุก route บน viewport มือถือ (390×844) ต่อ DB จริง + เช็คโค้ดฝั่ง API/หน้าประกอบทุกจุดที่สะดุด เพื่อแยกให้ชัดว่าอะไรคือบั๊ก อะไรคือ hardcode อะไรคือช่องว่างข้อมูล

**สรุปสั้น:** โครงหลักแข็งแรง — design system สม่ำเสมอมาก, CRUD หลัก (งาน/กิจกรรม/แจ้งซ่อม/ของหาย) ทำงานจริงครบ, สองภาษาครบ, dark theme ใช้ได้. แต่มี **ปัญหาใหญ่ 1 เรื่องที่จะทำให้ demo วันจริงดูพัง** (ข้อมูล import ผิดช่วงเวลา) และ **ค่าปลอม hardcode หลายจุดที่ขัดแย้งกับข้อมูลจริงบนหน้าเดียวกัน** ซึ่งผู้ใช้ครั้งแรกเห็นแล้วเสียความเชื่อถือทันที

## 🔴 P1 — ต้องแก้ก่อนนำเสนอ (demo จะดูพังถ้าไม่แก้)

### 1. ข้อมูล import ทั้งหมดอยู่ผิดช่วงเวลา → หน้า time-based ว่างเปล่าหมด
ตรวจจาก API จริง:
- `ClassPeriod` — มีแค่สัปดาห์ **22–26 มิ.ย.** (17 แถว)
- `MealItem` — มีแค่ **พ.ค.** (ก.ค. = 0 แถว)
- `AcademicCalendarEvent` — ทั้ง 31 แถวอยู่ **พ.ค.** ทั้งหมด

ผลที่ผู้ใช้เห็นวันนี้ (และจะเห็นวันนำเสนอ 10 ส.ค. ถ้าไม่แก้):
- หน้าแรก: "ไม่มีคาบเรียนวันนี้" + "ยังไม่มีเมนู"
- ปฏิทิน: เดือนปัจจุบันโล่ง 0 events ทุกวัน
- เมนูอาหาร: ขีด "—" ทั้ง 3 มื้อทุกวัน ไม่มีคำอธิบาย — ดูเหมือนแอปพัง ไม่ใช่ไม่มีข้อมูล

**แก้:** เตรียม CSV ช่วง ส.ค. (อย่างน้อยสัปดาห์ 10 ส.ค.) แล้วรัน `import:all` ใหม่ก่อนวันจริง — สำคัญที่สุดในไฟล์นี้

### 2. กระดิ่งแจ้งเตือนโชว์ "3" แต่หน้าแจ้งเตือนว่าง 0 รายการ
`unread={3}` เขียนตายใน `src/app/page.tsx:34`. ผู้ใช้กดตามตัวเลขแล้วเจอหน้าว่าง = ขัดแย้งจะๆ. Notification model มีใน schema แต่ไม่มีอะไรสร้าง notification จริงเลย.
**แก้:** ง่ายสุด = เอา badge ออก (`unread={0}`). ดีกว่า = seed notification จริง 2-3 อันให้ตรงกัน.

### 3. ตัวเลขปลอมใน "บริการยอดนิยม" ขัดกับข้อมูลจริง
`src/app/service/page.tsx:31-34` hardcode: "GPAX 3.62" (จริง 3.65 — ขัดกับหน้าโปรไฟล์ที่อยู่ห่างกันแค่คลิกเดียว), "5 กิจกรรม" (จริง 2), "แจ้งซ่อม 1 รายการ". ส่วน "ใช้ล่าสุด Recent" ก็ static — ผู้ใช้ครั้งแรกยังไม่เคยใช้อะไรแต่มีประวัติ "ล่าสุด" โผล่มา.
**แก้:** ตัดตัวเลข sub ออกเหลือแค่ชื่อบริการ (เร็วสุด ปลอดภัยสุด) หรือดึงจริงจาก API.

### 4. ป้าย "อยู่ในคาบเรียน" บนแบนเนอร์โปรไฟล์ static
ขัดกับการ์ด "ไม่มีคาบเรียนวันนี้" ที่อยู่ถัดลงมาบนหน้าแรกหน้าเดียวกัน (และวันอาทิตย์ก็ขึ้นว่าอยู่ในคาบ).
**แก้:** เอาออก หรือคำนวณจาก `/api/home` nextClass (มีข้อมูลพร้อมแล้ว).

## 🟠 P2 — ข้อมูลหลังบ้านทับซ้อน/ไม่เชื่อมกัน (ประเด็นที่ถามมาตรงๆ)

### 5. `/class` ไม่ได้ใช้ DB เลย — สอง source ซ้ำซ้อนกันจริง
หน้าตารางเรียนอ่าน mock จาก `src/lib/data/class.ts` ทั้งก้อน (cohort hardcode `"วก."` — เลยขึ้นหัว "วิศวกรรมเครื่องกล" ทั้งที่ user เป็น นนร.ปี 1) ขณะที่ `/api/class` + ข้อมูล ClassPeriod 17 แถวที่ import ไว้ **ไม่มีหน้าไหนเรียกใช้เลย** (orphaned). สรุป: ตารางเรียนมี 2 ชุดคนละเนื้อหา — หน้าโชว์ mock, DB มีของจริงแต่ไม่ถูกใช้, ส่วนการ์ด "คาบเรียนถัดไป" บนหน้าแรกใช้ DB → สองหน้านี้จึงเล่าเรื่องคนละชุดกันได้
**แก้:** ต่อหน้า class เข้า `/api/class?dayTh=` (API พร้อมแล้ว โครงหน้าเดิมใช้ได้) แล้วลบ `lib/data/class.ts`

### 6. Announcements/News — mock ทั้งระบบ ทั้งที่ schema มี model รออยู่แล้ว
`Announcement` + `NewsItem` มีใน Prisma schema แต่ไม่มี API route ไม่มี seed. หน้า announcements ×2, hero carousel, news feed อ่านจาก `lib/data/announcements.ts`. ผลข้างเคียงที่ผู้ใช้เห็น: timestamp ปลอมค้าง ("2 ชม. ที่แล้ว" ตลอดกาล), ประกาศลงวันที่ 9 มิ.ย. ผ่านมาเดือนกว่ายังติด badge "NEW"
**แก้ (ถ้าทัน):** API + seed สั้นๆ ประกาศจริง 4-5 อัน. **แก้ (ถ้าไม่ทัน):** อัปเดต mock ให้วันที่เป็นช่วงต้น ส.ค. ก่อน present กันดูเก่า

### 7. Service items ใน DB ก็ orphaned
seed มี ServiceCategory/ServiceItem (สวัสดิการ/ข้อมูลการศึกษา/ช่องทางติดต่อ) แต่หน้า service hardcode รายการคนละชุด. เลือกทางเดียว: ต่อ DB หรือลบ seed ส่วนนี้ทิ้งกันสับสน

### 8. `lib/data/todo.ts` = ไฟล์ตาย
ไม่มีใคร import แล้ว (หน้า todo ใช้ API จริงแล้ว) — ลบได้เลย

## 🟡 P2 — UX ที่ควรแก้

### 9. Lost & Found ไม่มีตัวเลือก "ของหาย" หรือ "ของพบ"
ฟอร์มชื่อ "แจ้งของพบ / ของหาย" แต่ไม่มีช่องให้เลือกว่าอันไหน — ทุกรายการกลายเป็นก้อนเดียวแยกไม่ได้ และปุ่มในหน้ารายละเอียดมีแค่ "นี่คือของฉัน → ขอรับคืน" ซึ่ง make sense เฉพาะของพบ (คนแจ้งของหายจะ claim ของตัวเองทำไม). สาเหตุ: schema ไม่เคยมี field `type` (บั๊กชุดที่แก้ไปรอบก่อนคือ UI อ้าง field นี้ทั้งที่ไม่มีจริง — ตอนนั้นแก้โดยตัด UI ให้ตรง schema, ตอนนี้ควรแก้ฝั่ง schema ให้ตรง product)
**แก้:** เพิ่ม enum `LostFoundType { lost, found }` + migration + radio 2 ปุ่มในฟอร์ม + label ปุ่ม claim ตาม type ("ฉันพบของชิ้นนี้" สำหรับ lost / "นี่คือของฉัน" สำหรับ found)

### 10. ฟอร์มแจ้งซ่อม: ช่อง "ส่งให้ทีม" ไม่ควรให้ นนร. เลือก
เป็น logic หลังบ้าน (routing งานให้ทีมช่าง) — ผู้ใช้ครั้งแรกไม่รู้จะเลือกอะไร เลือกผิดงานก็ไปผิดทีม. **แก้:** ตัด dropdown นี้ออกจากฟอร์มผู้ใช้ ให้เจ้าหน้าที่ assign ทีหลัง (field ใน DB มีอยู่แล้ว ไม่ต้องแตะ schema)

### 11. ฟอร์มแจ้งซ่อม: "รายละเอียด" ไม่มีดาว * แต่ API บังคับ
กรอกแค่หัวข้อแล้วกดส่งได้ → เจอ error ดิบภาษาอังกฤษ "titleTh and descriptionTh required". **แก้:** ใส่ `*` + disable ปุ่มจนกว่าจะกรอก (แบบเดียวกับฟอร์ม lost-found ที่ทำถูกแล้ว)

### 12. บั๊กวันหยุดสุดสัปดาห์ ×2
- **ตารางเรียน:** วันอาทิตย์ default ไปวันจันทร์ แล้ว badge "วันนี้" ขึ้นบนวันจันทร์ด้วย (`JS_DAY_MAP[getDay()] ?? "Monday"` ใช้ fallback เดียวกันทั้ง selectedDay และ isToday เลยเท่ากันเสมอ)
- **เมนูอาหาร:** วันอาทิตย์โชว์ "สัปดาห์นี้" เป็นสัปดาห์ที่แล้ว (13-17) แทนที่จะเป็นสัปดาห์ถัดไป
วันนำเสนอ 10 ส.ค. เป็นวันจันทร์ — ไม่โดนตรงๆ แต่ถ้าซ้อมวันอาทิตย์จะเจอ

### 13. สองฟีเจอร์พี่น้องกันแต่ pattern กลับด้าน
Lost & Found = เข้าแล้วเจอ **list** ก่อน มีปุ่ม + ไปฟอร์ม. แจ้งซ่อม = เข้าแล้วเจอ **ฟอร์ม** ก่อน มีลิงก์เล็กๆ "รายการของฉัน" มุมขวา. ควรเลือก pattern เดียว (แนะนำ list-first เหมือน lost-found — ผู้ใช้ส่วนใหญ่กลับมาดูสถานะบ่อยกว่าแจ้งใหม่)

### 14. Settings มีของค้าง/ตาย
- "เชื่อมต่อ LINE — เร็วๆ นี้" ทั้งที่บน production ผู้ใช้ login ผ่าน LINE อยู่แล้ว และ `/api/me` ส่ง `lineLinked` มาให้แล้วด้วย → ควรโชว์ "เชื่อมต่อแล้ว ✓" จริงๆ
- แถว "นโยบาย PDPA →" กดไม่ได้ (ไม่ใช่ลิงก์ เป็น div เฉยๆ) — PDPA เป็น requirement ในขอบเขตวิจัยด้วย ควรมีหน้าจริงหรือ bottom-sheet สั้นๆ

### 15. ลงทะเบียนกิจกรรมแล้ว ยกเลิกไม่ได้
API `/api/activity/[id]/rsvp` รองรับ `action: "cancel"` อยู่แล้ว แต่ UI หลังลงทะเบียนโชว์ "✓ ลงทะเบียนแล้ว" เฉยๆ ไม่มีปุ่มยกเลิก

### 16. Dropdown เรียง "อื่น ๆ" ไว้กลางรายการ
ทั้งฟอร์มแจ้งซ่อมและของหาย (เรียงตามอักษรไทย) — ธรรมเนียมคือ "อื่น ๆ" อยู่ท้ายสุดเสมอ

## ✅ จุดแข็งที่เห็นชัด (คงไว้แบบนี้)

- Design system แน่นมาก: สี/radius/typography เดียวกันทุกหน้า สแกนตาแล้วรู้ทันทีว่าแอปเดียวกัน
- Loading / error + retry / empty states มีเกือบครบทุกหน้า
- สองภาษาไทย/EN สม่ำเสมอ, dark theme ทำงานจริง
- Flow หลักที่ demo ได้จริง: งาน (สร้าง/ติ๊กเสร็จ), กิจกรรม (สร้าง+รูป+RSVP), แจ้งซ่อม (ฟอร์ม+รูป+หมุดแผนที่+ประวัติสถานะ), ของหาย (แจ้ง+รูป+claim), ผลการเรียน (GPA/GPAX คำนวณจริงจาก DB)
- Image compression ก่อนอัปโหลด — เร็วขึ้นจริงบนมือถือ
